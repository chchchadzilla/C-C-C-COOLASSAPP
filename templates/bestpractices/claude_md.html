📄 CLAUDE.md Mastery

The single most impactful file for agent quality. One agent with bad context is annoying — ten agents with bad context is a disaster.

← All Best Practices

What is CLAUDE.md?

CLAUDE.md is a special file that Claude Code automatically reads at the start of every session. Think of it as your agent's "briefing document" — it tells Claude about your project, coding standards, important conventions, and things to watch out for.

When running 10 concurrent agents, CLAUDE.md becomes the single source of truth that keeps all agents aligned. Without it, agents make contradictory decisions.

📂 File Hierarchy

Claude Code loads CLAUDE.md files at three levels, each adding more specific context:

~/.claude/CLAUDE.md Global — applies to ALL your projects (personal preferences, general style)
./CLAUDE.md Project root — project-wide standards, tech stack, architecture decisions
./src/CLAUDE.md Directory-level — only loaded when agent works in that directory
Pro tip: Directory-level CLAUDE.md files are only loaded when the agent touches files in that directory. Use them for module-specific conventions like "All API routes in this folder use Zod validation" or "Tests here use vitest, not jest."

📝 Production CLAUDE.md Template

CLAUDE.md — Copy this to your project root 
# CLAUDE.md

## Project Overview
[Project name] is a [type of project] built with [tech stack].
The main entry point is [file]. The project structure follows [pattern].

## Build & Test Commands
- Build: `npm run build`
- Test all: `npm test`
- Test single: `npm test -- --grep "test name"`
- Lint: `npm run lint`
- Dev server: `npm run dev`
- Type check: `npx tsc --noEmit`

## Code Style & Conventions
- Use TypeScript strict mode
- Prefer `const` over `let`, never use `var`
- Use named exports, not default exports
- Error handling: always use custom error classes from `src/errors/`
- Logging: use the logger from `src/utils/logger.ts`, never `console.log`

## Architecture Decisions
- API routes go in `src/routes/` with corresponding controller in `src/controllers/`
- Database access through repository pattern in `src/repositories/`
- All business logic lives in `src/services/`, never in controllers
- Use dependency injection via constructor parameters

## Common Gotchas
- The `user.id` field is a UUID string, not a number
- Environment variables are validated at startup in `src/config.ts` — add new ones there
- The `auth` middleware must come before `rateLimiter` in the middleware chain
- Database migrations run automatically on deploy — create new ones with `npm run migration:create`

## Testing Standards
- Every new function needs a unit test
- Integration tests go in `tests/integration/`
- Use `factories/` for test data, never hardcode
- Mock external services, never call real APIs in tests
- After making changes, ALWAYS run the relevant test suite and fix failures

## What NOT to Do
- Don't modify `src/generated/` files — they are auto-generated
- Don't add dependencies without checking bundle size impact
- Don't change the database schema without a migration
- Don't commit `.env` files

🔀 Multi-Agent CLAUDE.md Standards

When running multiple agents on the same codebase, add these sections to prevent conflicts:

Additional sections for multi-agent CLAUDE.md 
## Agent Coordination Rules
- ALWAYS create a new branch before making changes
- Branch naming: `agent/[type]-[task-description]` (e.g., `agent/feat-add-user-auth`)
- Never push directly to `main` or `develop`
- If you encounter a merge conflict, STOP and report it — do not auto-resolve
- Check `git status` before and after completing a task

## File Ownership Boundaries
- `src/auth/` — Only auth-related agents should modify these
- `src/api/` — API agents only; coordinate with auth changes
- `src/shared/` — DANGER ZONE: changes here affect everyone. Minimize modifications.
- `package.json` — Never add dependencies without explicit instruction

## Verification Protocol
After EVERY change:
1. Run `npm run typecheck` — fix any type errors
2. Run `npm test` — fix any test failures  
3. Run `npm run lint` — fix any lint errors
4. If all pass, commit with a descriptive message
5. If any fail, fix the issue before moving on

❌ Common CLAUDE.md Mistakes

Too vague

❌ "Follow good coding practices"
✅ "Use TypeScript strict mode. All functions must have explicit return types. Use Zod for runtime validation."

No build commands

❌ Assuming the agent knows how to build your project
✅ List every command: build, test, lint, format, deploy

Missing gotchas

❌ Not mentioning that `user.id` is a UUID, causing subtle bugs
✅ Document every non-obvious behavior and type surprise

Stale context

❌ CLAUDE.md references files or patterns that no longer exist
✅ Review and update CLAUDE.md weekly. Treat it as living documentation.

Too long

❌ 500+ line CLAUDE.md that wastes context window
✅ Keep it focused. Use directory-level files for module-specific rules. Use #tags for conditional loading.

No negative instructions

❌ Only telling the agent what TO do
✅ "What NOT to Do" section prevents expensive mistakes

💡 Pro Tips

Use #tags for conditional loading: You can add # Tags: frontend at the top, and agents can be instructed to only follow rules matching their domain.
Include a "decision log": When you make architecture decisions, document WHY in CLAUDE.md. This prevents agents from "helpfully" refactoring things back.
Version your CLAUDE.md: Track it in git. Review changes in PRs. When context goes wrong for multiple agents, `git blame` the CLAUDE.md.
Let Claude write it: Run claude "Generate a CLAUDE.md for this project by analyzing the codebase" — then review and refine.
Test with a new session: Start a fresh Claude Code session and give it a task. If it makes mistakes your CLAUDE.md should prevent, improve the file.
