🌈 The Multi-Agent Spectrum

Not all parallel work is the same. Understanding where your task falls on this spectrum determines which pattern to use:

1. Independent Tasks

Agents work on completely separate files/features. Zero coordination needed.

Example: Agent A writes docs, Agent B adds tests for module X, Agent C refactors module Y

2. Loosely Coupled

Agents work on related features that share interfaces. Need agreed-upon contracts.

Example: Agent A builds the API endpoint, Agent B builds the frontend that calls it

3. Tightly Coupled

Agents work on interrelated code. Requires orchestrator pattern.

Example: Refactoring a shared module while others depend on it

4. Sequential

Output of one agent is input of next. Pipeline pattern.

Example: Agent A designs schema → Agent B implements models → Agent C writes tests

🌿 The Git Worktree Pattern (Recommended)

This is the #1 pattern for parallel agents. Each agent gets its own working directory via git worktrees, eliminating file conflicts entirely.

Setup worktrees for parallel agents
# Create worktrees for each agent task
git worktree add ../project-auth   feature/auth-system
git worktree add ../project-api    feature/api-endpoints
git worktree add ../project-tests  feature/test-coverage
git worktree add ../project-docs   feature/documentation

# Launch agents in their own worktrees
cd ../project-auth  && claude "Implement JWT auth in src/auth/" &
cd ../project-api   && claude "Add CRUD endpoints for users" &
cd ../project-tests && claude "Add tests for src/services/" &
cd ../project-docs  && claude "Generate API docs from code" &

# When done, merge back
git merge feature/auth-system
git merge feature/api-endpoints
# etc.
Why this works: Each worktree is a real checkout with its own working directory. Agents can't accidentally step on each other's files. Merging happens explicitly under your control.

📡 The Fan-Out / Fan-In Pattern

Use Claude's built-in subagent capability to automatically parallelize work:

Fan-out prompt for Claude
# This single prompt creates parallel subagents automatically:
claude "Refactor the following modules to use the new error handling pattern.
Each module can be done independently:
1. src/services/auth.ts
2. src/services/billing.ts  
3. src/services/notifications.ts
4. src/services/analytics.ts

Use subagents to do these in parallel. Each subagent should:
- Read the existing module
- Apply the ErrorResult pattern from src/shared/errors.ts
- Update tests
- Verify with: npm test -- --grep [module-name]"
How it works: When Claude sees parallelizable work, it can spawn subagents using Task() tool calls. Each subagent runs independently, and results fan back to the orchestrator agent.

🖥️ Headless / Background Sessions

Run agents in the background without a terminal attached:

Background agent patterns
# Run in headless mode with a prompt
claude -p "Add input validation to all API endpoints" --output-format json

# Print-mode for scripting (no interactive session)
claude -p "Analyze this codebase and list all TODO comments" --output-format text

# Pipe input for batch processing
echo "Review src/auth/login.ts for security issues" | claude -p --output-format json

# Run with specific model
claude --model claude-sonnet-4-20250514 -p "Quick fix: update the README version to 2.0"

# Continue a previous session
claude --continue        # Resume most recent session
claude --resume [id]     # Resume specific session

🔒 Session Isolation Rules

✅ MUST

  • Each agent gets its own branch
  • Each agent works in its own worktree OR on non-overlapping files
  • Define clear file ownership boundaries in CLAUDE.md
  • Use verification commands after every change
  • Commit frequently with descriptive messages

❌ NEVER

  • Two agents on the same branch
  • Two agents modifying the same file simultaneously
  • Agents resolving merge conflicts autonomously
  • Agents pushing to main/develop directly
  • Running agents without a CLAUDE.md in the project

💡 SHOULD

  • Name branches with agent identifiers
  • Set up branch protection rules
  • Review agent PRs before merging
  • Start with 2-3 agents, scale up gradually
  • Monitor token usage across all sessions

📈 Recommended Scaling Schedule

Don't go from 1 to 10 overnight. This ramp prevents cascading failures:

1-2

Days 1-4: Single Agent Mastery

Use 1 agent. Perfect your CLAUDE.md. Establish verification habits.

3

Day 5: First Parallel Exercise

3 agents on completely independent tasks. Watch for any file overlap.

5-7

Days 6-8: Scale with Guardrails

5-7 agents. Introduce worktrees. Identify your failure modes.

10

Days 9-14: Full Fleet

10 agents with orchestration layer. Automated health checks. Nightly reports.
